# 1. Overview

This document defines the MongoDB database schema used by DebateAI.

The database is responsible for storing:

* User accounts
* Debate sessions
* Debate messages
* Agent configurations
* User preferences
* Future analytics

The design follows MongoDB best practices by storing related data together where appropriate while avoiding oversized documents.

## 6. messages Collection

### Purpose

Stores every message generated during a debate.

Each document represents a single message generated by either the user or an AI agent.

This design makes it easy to:

* Add new AI providers
* Support multiple models
* Track performance
* Analyze token usage
* Compare model outputs

### Example Document

```json
{
  "_id": ObjectId("..."),

  "debate_id": ObjectId("..."),

  "round": 1,

  "role": "assistant",

  "agent": "openai_agent",

  "provider": "openai",

  "model": "gpt-4.1",

  "message_type": "initial_response",

  "content": "AI should assist doctors rather than replace them.",

  "tokens": {
    "input": 456,
    "output": 189,
    "total": 645
  },

  "latency_ms": 2140,

  "cost": 0.0028,

  "status": "completed",

  "created_at": "2026-07-15T12:30:10Z"
}
```

## 8. Agent Fields

### agent

Represents the logical role of the participant.

Possible values

```text
user

openai_agent

gemini_agent

consensus_agent

research_agent

fact_checker

planner

reviewer
```

The agent field should describe the role performed during the debate rather than the underlying model.

---

### provider

Represents the AI service provider.

Examples

```text
openai

google

anthropic

groq

openrouter

together

cohere
```

This makes it possible to swap providers without changing application logic.

---

### model

Stores the exact model used for that response.

Examples

```text
gpt-4.1

gpt-4o-mini

gemini-2.5-flash

gemini-2.5-pro

claude-sonnet-4

llama-3.3-70b

qwen3
```

This enables:

* Model comparison
* Cost analysis
* Performance benchmarking
* A/B testing
* Analytics dashboards

## 16. Analytics (Future)

With the additional fields (`provider`, `model`, `cost`, and `latency_ms`), DebateAI can generate rich analytics such as:

* Total debates per user
* Total token consumption
* API cost per debate
* Average latency by provider
* Average latency by model
* Fastest model
* Most frequently used model
* Cost comparison across providers
* Debate completion rate
* Model win rate (if winner selection is introduced)
* Consensus generation time
* Daily and monthly usage statistics

This schema allows new providers and models to be added without any database redesign, making the system extensible for future AI capabilities.
